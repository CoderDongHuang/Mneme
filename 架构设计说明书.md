# Mneme - 三级记忆个人学习助手 Agent 架构设计说明书

## 一、项目目标

### 核心目标
构建一个具备**三级记忆架构**的个人学习助手 Agent，用户上传课件/笔记后，Agent 能基于资料回答问题，并通过记忆系统记住用户的学习偏好、知识薄弱点、学习进度，提供个性化学习建议。

### 一句话定位
> Agent 不只是一个问答机器，它会"记住你、了解你、主动帮你"。

## 二、技术栈

### 已确定（需求文档明确）

| 层级 | 技术 | 用途 |
|------|------|------|
| Python 层 | LangGraph | 状态图编排，Agent 决策链 |
| Python 层 | LangChain | Tool 封装与 Chain 编排 |
| Python 层 | Chroma | 向量存储与语义检索 |
| Python 层 | FastAPI | 暴露 Agent 推理接口 |
| Python 层 | DeepSeek-V3 API | 对话/意图/摘要/记忆提取 |
| Python 层 | 通义千问 text-embedding-v3 | 文档与记忆向量化 |
| Python 层 | APScheduler | 轻量调度，管理记忆反思触发 |
| Java 层 | Spring Boot 3.x | 用户认证、多租户隔离、请求调度 |
| Java 层 | MyBatis Plus | 知识库元数据管理 |
| Java 层 | WebSocket | Agent 回复实时推送 |
| Java 层 | Redis | 热点缓存 |
| Java 层 | MySQL | 用户/知识库/配置存储 |
| 部署 | Docker | 容器化部署 |

### 已确认的择优选择

| 选项 | 选择 | 说明 |
|------|------|------|
| Embedding 模型 | 通义千问 text-embedding-v3 | 国内直连，免费额度充裕 |
| LLM 模型 | DeepSeek-V3 | 国内直连，推理能力强，结构化任务表现好 |
| 前端 | 暂不开发，用 Swagger 调试 | 面试展示用录屏 |
| 文档解析 | LangChain Document Loaders | 和 LangChain 生态统一 |
| 分块策略 | 语义分块（LangChain SemanticChunker） | 语义分块效果更好 |

## 四、开发规范

### 4.1 Java 层规范

| 规范项 | 要求 |
|--------|------|
| 命名 | 类名大驼峰，方法/变量小驼峰，常量全大写下划线 |
| 分层 | Controller → Service → Mapper，禁止跨层调用 |
| 接口返回 | 统一 `Result<T>` 包装，包含 code/message/data |
| 异常处理 | 全局 `@RestControllerAdvice` 统一异常拦截 |
| 日志 | SLF4J + Logback，关键操作记录 INFO，错误记录 ERROR |
| 事务 | Service 层使用 `@Transactional`，Controller 层不开事务 |

### 4.2 Python 层规范

| 规范项 | 要求 |
|--------|------|
| 命名 | 模块/函数/变量蛇形命名，类名大驼峰 |
| 类型注解 | 所有函数签名必须带类型注解 |
| 数据结构 | 统一使用 Pydantic BaseModel 定义请求/响应模型 |
| 异步 | I/O 操作使用 async/await，禁止阻塞主线程 |
| 日志 | logging 模块，按模块分 logger，JSON 格式输出 |
| 错误处理 | 自定义异常类，FastAPI 统一异常处理器 |

### 5.2 各层职责与边界

#### Java 层

| 模块 | 职责 | 边界 | 不负责 |
|------|------|------|--------|
| Controller | 接收 HTTP 请求，参数校验，返回响应 | 只做请求转发和校验 | 不写业务逻辑 |
| Service | 业务逻辑处理，调用 Python Agent，缓存管理 | 编排业务流程 | 不直接操作数据库 |
| Mapper | 数据库 CRUD | 单表操作 | 不写复杂业务逻辑 |
| WebSocket | 实时推送 Agent 回复 | 连接管理和消息推送 | 不处理 AI 逻辑 |

#### Python 层

| 模块 | 职责 | 边界 | 不负责 |
|------|------|------|--------|
| API 路由 | 接收 Java 请求，返回 JSON 响应 | 请求解析和响应格式化 | 不写业务逻辑 |
| Agent 图 | LangGraph 状态图编排，决策链执行 | AI 推理流程 | 不管理用户/权限 |
| 记忆系统 | 三级记忆的读写、蒸馏、反思 | 记忆生命周期管理，含 APScheduler 调度 | 不处理知识库文档 |
| 知识库模块 | 文档解析、切分、向量化、检索 | 知识数据处理 | 不管理用户会话 |
| Tool 层 | 封装知识库查询、记忆查询为 LangChain Tool | 工具接口标准化 | 不决定何时调用 |

## 八、意图识别设计

### 8.1 意图分类

用户请求进入 LangGraph 状态图后，第一个节点进行意图识别，分为三类：

| 意图类型 | 触发场景 | 处理链路 |
|----------|----------|----------|
| `qa` | 基于知识库的问答，如"反向传播是什么？" | 检索知识库 → LLM 推理 → 返回答案 |
| `review` | 回顾历史内容，如"上次那个公式再讲一下" | 检索记忆 → 召回历史对话 → LLM 推理 |
| `suggest` | 请求学习建议，如"我接下来该学什么？" | 检索薄弱点 → 检索关联知识 → 生成建议 |

### 8.2 状态图分支

```
用户请求
  │
  ▼
[意图识别节点]
  │
  ├── intent=qa ────→ [知识库检索节点] ──→ [LLM 推理节点] ──┐
  │                                                          │
  ├── intent=review ─→ [记忆检索节点] ────→ [LLM 推理节点] ──┤
  │                                                          │
  └── intent=suggest → [薄弱点检索节点] ─→ [建议生成节点] ───┤
                                                             │
                                                             ▼
                                                      [格式化响应节点]
                                                             │
                                                             ▼
                                                  [记忆蒸馏节点]（会话结束时触发）
```

### 8.3 意图识别实现

在 `agents/nodes.py` 中，使用 LLM 进行轻量级意图分类（非 Tool Calling），输出 JSON 格式：

```json
{
  "intent": "qa",
  "confidence": 0.92,
  "extracted_entities": ["反向传播"]
}
```

当 `confidence < 0.7` 时，默认走 `qa` 链路，并在响应中追加澄清追问。

### 8.4 记忆检索策略

不同意图检索不同类型的记忆，避免无效检索消耗 token：

| 意图 | 检索的记忆类型 | 说明 |
|------|----------------|------|
| `qa` | 用户偏好 | 仅检索偏好记忆，影响回答风格（如"喜欢图表解释"） |
| `review` | 薄弱点 + 历史对话 | 检索相关薄弱点和历史对话片段 |
| `suggest` | 全部记忆 | 检索偏好 + 薄弱点 + 学习进度，综合生成建议 |

## 九、文件上传流程

### 9.1 上传链路

```
前端上传文件
  │
  ▼
Java Controller 接收文件
  │
  ▼
Java Service 存储到共享目录 /data/files/{user_id}/{kb_id}/
  │
  ▼
Java Service 调用 Python POST /api/v1/knowledge/ingest，传入 file_path
  │
  ▼
Python 读取文件 → 解析 → 切分 → 向量化 → 存入 Chroma
  │
  ▼
Python 返回处理结果，Java 更新 MySQL 文档状态
```

### 9.2 共享存储

`docker-compose.yml` 配置共享 volume：

```yaml
volumes:
  - ./data:/data
```

Java 和 Python 容器都挂载 `/data`，Java 写文件、Python 读文件，通过路径通信。

### 9.3 Chroma 持久化

Chroma 使用 `PersistentClient`，数据目录映射到宿主机，容器重启不丢失：

```yaml
volumes:
  - ./data/chroma:/chroma/data
```

Python 代码中：
```python
client = chromadb.PersistentClient(path="/chroma/data")
```

## 十二、Chroma 多租户隔离设计

### 12.1 Collection 命名规则

Chroma 本身没有用户隔离机制，所有 collection 共享在同一实例中。通过命名规则实现多租户隔离：

```
collection_name = f"user_{user_id}_kb_{kb_id}"
```

示例：`user_1001_kb_2001`

### 12.2 统一管理函数

在 `knowledge/vector_store.py` 中封装 collection 操作，禁止直接调用 Chroma 原生 API：

```python
def _get_collection_name(user_id: str, kb_id: str) -> str:
    """统一 collection 命名，格式: {user_id}_{kb_id}"""
    return f"user_{user_id}_kb_{kb_id}"

def get_or_create_collection(user_id: str, kb_id: str) -> chromadb.Collection:
    """获取或创建 collection，自动应用命名规则"""
    ...

def delete_collection(user_id: str, kb_id: str) -> None:
    """删除 collection，用户删除知识库时调用"""
    ...
```

### 12.3 安全边界

- Java 层负责用户鉴权，确保用户只能访问自己的知识库
- Python 层信任 Java 传入的 `user_id`，不做二次鉴权
- 删除知识库时，Java 层先调 Python 删除 Chroma collection，再删 MySQL 记录

## 十四、关键设计决策记录

| 决策点 | 选择 | 理由 |
|--------|------|------|
| Java/Python 分离 | 是 | 职责清晰，Java 擅长基建，Python 擅长 AI |
| 通信方式 | HTTP | 简单可靠，WebSocket 仅用于推送 |
| 向量数据库 | Chroma | 轻量，适合个人项目，易部署 |
| 记忆存储 | Chroma（向量）+ 内存（上下文） | 语义检索用向量，短期用内存 |
| 前端 | 延后 | 优先验证核心功能 |
| 记忆反思触发 | Python APScheduler | 反思逻辑在 Python 层，由 Python 自己判断何时触发 |
| Chroma 多租户 | 命名规则隔离 | `{user_id}_{kb_id}` 格式，统一管理函数封装 |
| 意图识别 | LLM 轻量分类 | 三类意图（qa/review/suggest），低置信度时澄清追问 |
| 模型选型 | DeepSeek-V3 + 通义千问 Embedding | 国内直连零门槛，学生开发成本低，无代理依赖 |

### 模型接入方式

| 模型 | LangChain 类 | 环境变量 | 说明 |
|------|-------------|----------|------|
| DeepSeek-V3 | ChatDeepSeek | DEEPSEEK_API_KEY | 对话/意图/摘要/记忆提取 |
| 通义千问 Embedding | DashScopeEmbeddings | DASHSCOPE_API_KEY | 文档与记忆向量化 |

两个 API Key 都在各自官网免费注册获取，国内直连，无需代理。

## 十五、异常处理策略

### 15.1 常见异常场景及处理

| 异常场景 | 处理层级 | 处理方式 | 用户感知 |
|----------|----------|----------|----------|
| Chroma 服务不可用 | Python | 捕获连接异常，返回降级响应："知识库暂时不可用，请稍后重试" | 友好提示 |
| LLM API 超时（>30s） | Python | 捕获超时异常，返回："回答生成超时，请简化问题或重试" | 友好提示 |
| 用户上传损坏的 PDF | Python | 解析失败时标记文档状态为 `failed`，返回具体错误信息 | "文档解析失败，请检查文件是否损坏" |
| Java 调用 Python 超时 | Java | 30s 超时后返回："AI 服务响应超时，请稍后重试"，记录 ERROR 日志 | 友好提示 |
| 用户无权限访问知识库 | Java | Controller 层鉴权拦截，返回 403 | "无权访问该知识库" |
| 意图识别置信度过低 | Python | 默认走 `qa` 链路，响应末尾追加澄清追问 | 正常回答 + "你是想了解 XX 吗？" |
| 文档含图片/表格/公式 | Python | 仅提取文本内容，图片/表格/公式跳过 | "文档解析完成，图片/表格内容暂不支持" |

### 15.2 错误响应格式

所有错误统一使用标准格式，前端/调用方可解析：

```json
{
  "code": 500,
  "message": "知识库暂时不可用，请稍后重试",
  "data": null,
  "error_type": "service_unavailable"
}
```

`error_type` 用于前端区分错误类型并展示不同提示：
- `service_unavailable`：服务不可用，建议稍后重试
- `timeout`：请求超时
- `invalid_input`：输入格式错误
- `permission_denied`：权限不足
- `internal_error`：内部错误，记录日志并提示联系管理员
